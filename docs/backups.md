# Бэкапы и восстановление

Production хранит финансовую историю заведений — резервное копирование это
условие эксплуатации, а не опция. Требование квартальной копии и контроля
восстановления входит и в регуляторные обязательства
([israel-compliance.md](israel-compliance.md), «Резервные копии»).

Прогон восстановления по этому runbook выполнен 18.07.2026 — журнал в конце
документа.

## Слои защиты

| Слой | RPO | Кто включает / выполняет |
|---|---|---|
| PITR (WAL-архивация) | минуты | владелец: Dashboard → Database → Backups → Point in Time; **основной механизм**, суточный бэкап теряет до дня продаж |
| Ежедневные managed-бэкапы | до суток | входят в платный план; наличие и срок хранения проверять в Dashboard → Database → Backups |
| Логический дамп `npm run db:dump` | на момент дампа | вручную: перед крупными миграциями и раз в квартал (квартальную копию — вне машины разработчика, зашифрованной) |
| Файлы Storage (фото меню, логотипы) | — | в pg-дамп **не входят**; см. «Storage» ниже |

## Логический дамп

```bash
npm run db:dump   # project-ref guard → roles.sql + schema.sql + data.sql в backups/<дата>/
```

Что внутри `data.sql`: схемы `auth` (устройства-аккаунты с паролями), `public`
(все доменные данные, включая счётчики и `setval` для sequences) и метаданные
`storage`. Дамп содержит PIN-хэши сотрудников и телефоны гостей — хранить
только зашифрованным, каталог `backups/` в `.gitignore`.

## Восстановление

### Сценарий А: порча/потеря данных, проект жив → PITR

1. Dashboard → Database → Backups → Point in Time → выбрать момент **до**
   инцидента. Восстановление откатывает весь проект целиком: заказы, пробитые
   после точки восстановления, будут потеряны — их пере-проводят по бумажным
   чекам компенсирующими операциями, а не повторным restore.
2. После восстановления обязательно: блок «Верификация» ниже, сверка последнего
   `receipt_number` с последним физически напечатанным чеком, полный
   [smoke-test T2](t2-smoke-test.md).

### Сценарий B: проект недоступен → новый проект + дамп

1. Создать новый Supabase-проект (тот же регион), `supabase link` на него.
2. Схему разворачивать **миграциями**, а не `schema.sql`:
   `supabase db push` — это тот же путь, каким живёт прод, он проверен CI и
   прогоном 18.07.2026. `schema.sql` из дампа — резерв на случай, если
   миграции недоступны.
3. Роли: `psql -f roles.sql` (ошибки `already exists` допустимы).
4. Данные — одной транзакцией, с отключёнными триггерами (иначе триггеры
   склада проведут списания второй раз) и **без** таблиц `storage.*`
   (их структура принадлежит Storage-сервису и отличается между версиями;
   бакеты создаются заново, см. «Storage»):

   ```bash
   awk '/^COPY "storage"/{skip=1} skip&&/^\\.$/{skip=0; next} !skip' data.sql > data_restore.sql
   psql "$NEW_DB_URL" --single-transaction -v ON_ERROR_STOP=1 \
     -c 'SET session_replication_role = replica' -f data_restore.sql
   ```

5. Edge Functions: `npm run functions:deploy`; секреты перенести в
   Supabase Secrets заново — в дампе их нет.
6. Frontend: у нового проекта новые URL и anon key → обновить
   `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` в Vercel и передеплоить.
   Обновить `EXPECTED` в `scripts/check-project-ref.mjs` и документацию.
7. Устройства: сохранённые сессии подписаны старым JWT-секретом — каждый
   терминал перелогинивается тем же email/паролем (пароли восстановлены
   вместе с `auth.users`).
8. Блок «Верификация», затем полный [smoke-test T2](t2-smoke-test.md).

### Верификация после восстановления

```sql
-- счётчик чеков непрерывен и совпадает с последним чеком
select (select counter from receipt_counters limit 1) as counter,
       (select max(receipt_number) from orders)        as max_receipt;
-- оплаченные заказы сходятся с платежами до агорота
select count(*) as mismatches from (
  select o.id from orders o
  left join payments p on p.order_id = o.id and p.refund_id is null
  where o.status in ('paid','refunded')
  group by o.id
  having o.total + coalesce(o.tip_amount,0) <> coalesce(sum(p.amount),0)
) m;
-- версия схемы соответствует фронту
select public.get_schema_version();
```

Дополнительно сверить количество строк ключевых таблиц (`orders`, `payments`,
`refunds`, `shifts`) с исходным дампом:
`awk '/^COPY /{t=$2;n=0;next} /^\\.$/{if(t!=""){print n,t;t=""};next} t!=""{n++}' data.sql`.

### Storage

`pg`-дамп переносит только метаданные Storage, не сами файлы. Фото меню и
логотипы — некритичные данные: после восстановления в новый проект бакет
`menu-images` создаётся заново, картинки перезаливаются вручную (старые
`image_url` в любом случае указывают на домен старого проекта). Если потеря
картинок неприемлема, снимать их копию отдельно (Storage API / dashboard)
вместе с квартальным дампом.

## Регламент

- **Квартально**: `npm run db:dump` → зашифрованная копия вне машины
  разработчика → строка в журнал ниже.
- **Ежегодно и перед крупными релизами**: прогон сценария B в чистый локальный
  стек (`supabase db reset` → restore дампа → верификация) — восстановление,
  которое ни разу не прогоняли, бэкапом не считается.
- **Перед крупными миграциями** (денежные таблицы, массовый backfill):
  свежий `npm run db:dump`.
