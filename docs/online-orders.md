# Онлайн-заказы с сайта (050–061)

Гость открывает публичную страницу, собирает корзину, оставляет имя и телефон,
выбирает доступный тип заказа: здесь, с собой или доставка. Касса получает
заявку (realtime + звонок), принимает или отклоняет. Принятая заявка становится
обычным заказом кассы с `source='site'`, попадает в очередь бариста и оплачивается
на кассе при получении. Фискальный контур не тронут: чек рождается в обычном
`pay_order`.

## Архитектура

```
Гость (телефон)                Supabase                       Касса (T2)
/order/<location_id>  ──POST──▶ Edge Fn public-order ──rpc──▶ online_orders (staging)
       │                          (service_role)                   │ realtime
       │◀─GET (поллинг 5с)─ Edge Fn public-order ◀─rpc─┐           ▼
       │                                               │      экран /online
/order меню ◀────GET──── Edge Fn public-menu           │      Принять ──▶ accept_online_order
                                                       │            └──▶ place_order (цены из
                                                       └──── статус       каталога, сток 047,
                                                                          очередь бариста)
```

Ключевые решения:

- **Анонимный гость ≠ устройство кассы.** Гость ходит только в Edge Functions
  (anon-ключ как bearer); прямого доступа к таблицам нет. Вся валидация, цены,
  идемпотентность и анти-спам — в `submit_online_order` (SECURITY DEFINER,
  EXECUTE только у service_role).
- **Часы приёма без настроек:** заявка принимается только при открытой смене
  на точке. Смена закрыта → гость видит «Сейчас закрыто».
- **Заявка ≠ заказ.** Снапшот позиций и цен хранится в `online_orders` для
  карточки кассира; настоящий заказ создаётся только при принятии, через
  `place_order` (цены пересчитываются из каталога заново, остатки списываются
  триггерами 047, стоп-лист уважается на этапе заявки).
- **Неоплаченный принятый заказ** (гость не пришёл) закрытие смены аннулирует
  как брошенный counter-заказ (035), остаток вернёт триггер 047.
- **Анти-спам в БД:** ≤3 заявок с одного телефона за 15 минут, ≤30
  необработанных заявок на точку, лимиты длины полей, ≤30 позиций.
- **client_uuid** генерирует страница гостя: ретрай POST не создаёт дубликат,
  он же — «секрет» для поллинга статуса и client_uuid будущего заказа.

   supabase functions deploy public-menu --project-ref qgmnxrgtlpyqglwqmsej

## Поведение кассы

- Новая заявка: звонок + тост на любом экране, счётчик на пункте «Онлайн».
- «Принять» — нужна открытая смена (иначе подсказка). Печатается кухонный
  тикет (если включён в настройках кассы), заказ виден бариста.
- «Отклонить» — с необязательной причиной, гость увидит её на странице.
- «Оплатить» при получении — обычный PaymentSheet (наличные/карта/кошельки,
  смешанная), чек — как всегда («Как выдать чек?» / автопечать).
- «Отменить» принятый (гость передумал) — право `void_order`, остаток вернётся.

## Self-service для любого клиента (051)

Настройки → Обслуживание → «Онлайн-заказы»: тумблер приёма (сервер
отклоняет заявки кодом `disabled`, витрина `public-menu` отдаёт
`accepting` — гостевые страницы прячут оформление заранее), готовая
ссылка `/order/<location_id>` с копированием и печать QR-флаера на
встроенном термопринтере. Владелец новой точки включает онлайн-заказы
сам, без разработчика: завёл меню → распечатал QR → повесил на стойку.

Там же — соцссылки (Instagram / Facebook / отзыв в Google): хранятся в
`locations.settings.online_orders`, `public-menu` отдаёт их в
`location.links`, гостевая страница показывает кнопки в подвале главного
экрана (пустая ссылка — кнопки нет).

## Пауза приёма и время приготовления (054, Square-стиль)

На экране «Онлайн» в шапке — две кликабельные пилюли (как «Orders on» и
«20 min prep time» в Square):

- **Статус приёма** (зелёная/жёлтая/серая точка) → шит с паузой на
  30 мин / 1 час / 2 часа / до конца дня. Пауза снимается сама —
  это метка `paused_until` в `locations.settings.online_orders`,
  сравнение по времени, без крона. Возобновить можно и вручную.
- **Время приготовления** → диапазон min–max или «Не показывать».
  Гость видит вилку времени, а после принятия — обратный отсчёт до верхней
  границы.

Пишут точечные RPC `set_online_pause` / `set_online_prep_range`
(миграции 054/061) — не `update_location_config`: тот требует `manage`,
а паузу должен уметь ставить и бариста. Право — `online_pause`
(Настройки → Права доступа), по умолчанию доступно всем.

`set_online_prep_minutes` оставлен как обратимо совместимая обёртка
`min=max`. Старый ключ `prep_minutes` также читается как диапазон из одного
значения.

Enforcement: `submit_online_order` отклоняет заявку кодом `paused`;
`public-menu` отдаёт `accepting` с учётом паузы + `paused_until`
(гость видит «приём приостановлен до 14:30») и `prep_min`/`prep_max`.
Порядок деплоя тот же: миграция → edge function `public-menu` → фронтенд.

## Типы заказа: здесь / с собой / доставка (058)

Гость выбирает тип заказа при оформлении; владелец включает варианты в
Настройки → Обслуживание → Онлайн-заказы → «Типы заказа» (тумблеры
здесь/с собой/доставка, хотя бы один обязателен). Хранится в
`locations.settings.online_orders.order_types` (массив). Отсутствие/пусто
= дефолт `['here','takeaway']` — старые точки без настройки продолжают
работать (раньше был хардкод `takeaway`).

- `public-menu` отдаёт `location.order_types`; гостевая страница
  показывает вопрос «Как хотите получить?» только если включён >1 типа
  (один — не спрашиваем). Доставка требует адрес (поле на чекауте).
- `public-order` прокидывает `order_type`/`delivery_address` в
  `submit_online_order` — тип валидируется по списку владельца
  (`invalid_order_type`), адрес обязателен для доставки
  (`invalid_address`). Хранятся в `online_orders.order_type` /
  `.delivery_address`.
- `accept_online_order` передаёт сохранённый тип в `place_order` (не
  хардкод) и копирует адрес в `orders.delivery_address`. Тип заказа и
  адрес видны кассиру на карточке заявки (`/online`).

**Порядок деплоя (как всегда):** миграция 058 → `public-menu` +
`public-order` → фронтенд.
