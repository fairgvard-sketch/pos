# Ручной smoke-test на T2 (Sunmi, Android 7.1)

Прогонять **на реальном T2** (или Sunmi-классе с Chrome < 80) перед выкатыванием
изменений, затрагивающих горячий кассовый поток, печать или совместимость движка.
Автотесты (`npm run test:run`) движок/принтер устройства не покрывают.

## 0. Предусловия

- APK-обёртка установлена и открывает прод-домен (`pos-self-sigma.vercel.app`).
- Встроенный принтер заправлен лентой 80 мм.
- Настройки → Устройство → «Движок браузера»: зафиксировать версию (Chrome NN).
  Предупреждение о Chrome < 80 видно только после успешного запуска UI.
- Legacy-бандл собирается для Chrome 52, но runtime-гейт отдельно требует
  Grid, CSS variables, Proxy, Map/Set и fetch. Chrome 57–83 без `flex-gap`
  должен открыться с CSS-fallback; stock Chrome 52 обычно покажет
  **диагностический экран** из-за отсутствия Grid.

## 1. Гейт совместимости (P2)

- [ ] Capability-screen либо честно перечисляет отсутствующие возможности,
      либо касса открывается без белого экрана.
- [ ] Если показан capability-screen, зафиксировать версию и остановить
      функциональный прогон до обновления WebView: остальные пункты
      на неподдержанном движке не считаются пройденными.
- [ ] Раскладка не «поехала»: сайдбар слева (LTR) / справа (RTL), плитки товаров
      в сетке, отступы ровные (проверка Grid/flex-gap на этом движке).
- [ ] Переключить язык he↔ru: направление RTL/LTR и отступы `ms/me` корректны
      (логические свойства — на старом движке проверять глазами, автопроверки нет).

## 2. Основной поток: PIN → товар → модификатор → оплата → чек

- [ ] `/pin`: ввод 4 цифр — автовход на 4-й цифре, ≤2 сек.
- [ ] `/sell`: тап по товару с модификаторами → выбрать модификатор → товар в корзине.
- [ ] Тап «Оплатить» → выбрать «Наличные» → подтвердить. Интерфейс НЕ висит
      (optimistic/outbox-first, не ждёт 4 сек).
- [ ] Автопечать чека: лента выезжает, чек читаемый, иврит/RTL корректны,
      суммы совпадают с экраном, `מקור` (оригинал) — один раз.
- [ ] Перепечать из «Операции» → чек помечен `העתק` (копия).

## 3. Печать: длинный чек (P6 — фикс обрезки canvas)

- [ ] Пробить заказ на **20+ позиций** (можно повторами) с модификаторами.
- [ ] Оплатить, распечатать. Чек НЕ обрезан — последняя строка (итог/подвал)
      на месте (раньше фиксированные 3000px обрезали хвост).
- [ ] Кухонный тикет длинного заказа тоже целиком.

## 4. Печать: результат задания (P6 — callback принтера)

- [ ] Отключить/зажевать ленту → пробить продажу с автопечатью.
- [ ] Появляется **ненавязчивое уведомление об ошибке печати** с кнопкой
      «Повторить», а НЕ тихая потеря. Второй экземпляр НЕ печатается, если
      первый упал.
- [ ] Заправить ленту → «Повторить» → чек печатается.

## 5. Офлайн-поток (P3/P4)

- [ ] Включить авиарежим. Пробить продажу наличными — временный чек `מסמך זמני`
      с локальным номером `K-n` печатается, интерфейс не висит.
- [ ] Бейдж офлайн-очереди в сайдбаре показывает количество.
- [ ] Вернуть сеть → очередь синхронизируется, тост `K-n → #NN · чек №…`.
- [ ] **Drain без PIN:** закрыть приложение с непустой очередью, где есть
      отмена/скидка позиции стола → открыть заново (PIN ещё не введён) →
      бейдж синий «введите PIN», очередь НЕ красная. Ввести PIN → очередь
      продолжается сама.

## 6. Лимит наличных (068 + frontend guard)

Большой чек собирать через «Свободная позиция» с крупной ценой.

- [ ] Чек ровно `6 000 ₪` — вся сумма наличными проходит без предупреждений.
- [ ] Чек `6 001 ₪` — подсказка «наличными не больше …» показывает `600,10 ₪`
      (меньшее из 10% и 6 000); попытка ввести наличными больше — блокируется
      с текстом `cashLimitError`, платёж НЕ создаётся.
- [ ] Смешанная оплата чека `> 6 000 ₪`: наличные в пределах лимита + остаток
      картой — проходит; наличная часть сверх лимита — блокируется.
- [ ] «Разделить поровну»: лимит наличных считается по всему чеку, а не по
      доле — вторая наличная доля сверх остатка лимита блокируется.
- [ ] Авиарежим: наличная оплата сверх лимита блокируется ДО записи в
      офлайн-очередь (бейдж очереди не увеличивается).
- [ ] Тексты лимита корректны на русском и иврите (переключить язык).

## 7. Отвязка устройства (P3)

- [ ] С непустой офлайн-очередью: Настройки → Устройство → «Отвязать» →
      диалог показывает **предупреждение о N неотправленных операциях**.
- [ ] После отвязки и входа под другим аккаунтом: чужой каталог/очередь
      предыдущего устройства НЕ видны; старые операции не уходят под новой
      сессией (карантин).

## Что зафиксировать в отчёте

- Версия WebView (Chrome NN), модель устройства.
- Любые «поехавшие» экраны со скриншотом (кандидаты на CSS-фолбэк).
- Печать: читаемость, обрезка, поведение при ошибке.

> Chrome 52 — цель транспиляции legacy-бандла, а не гарантия runtime-поддержки.
> Фактический минимум задаёт capability-check и подтверждает только этот ручной
> прогон на реальном устройстве.
