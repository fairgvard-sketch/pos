# Соответствие POS требованиям Израиля

Статус документа: рабочая инженерная матрица на 15 июля 2026 года.

Этот документ не является юридическим или налоговым заключением. Он переводит
официальные требования в задачи для продукта. До коммерческого запуска выводы,
типы документов, форматы печати и спорные сценарии должен письменно подтвердить
израильский `רואה חשבון` или `יועץ מס`, знакомый с регистрацией бухгалтерских
систем.

Нельзя заявлять, что Kassa «соответствует законодательству Израиля», пока не
выполнены release-gates ниже и не получено требуемое регистрационное
свидетельство.

## Реализуемое правило наличных

Для обычной сделки с бизнесом:

- до `6 000 ₪` включительно можно принять всю сумму наличными;
- выше `6 000 ₪` наличными можно принять не больше меньшего из `10%` суммы
  сделки и `6 000 ₪`.

В коде все расчёты выполняются в агоротах без float. Наличная часть суммируется
по всему чеку, включая смешанную оплату и «разделить поровну». `tendered` не
считается оплатой: юридический guard использует фактически зачтённый `amount`,
а сдача хранится отдельно.

Миграция `068_israel_cash_limit.sql`:

1. делает старую реализацию `pay_order_unchecked` недоступной клиенту;
2. проверяет все части до первой финансовой записи;
3. требует точного равенства суммы платежей долгу — переплата записывается как
   `tendered/change_due`, а не как завышенный `amount`;
4. сохраняет прежнюю атомарность и идемпотентность `p_payment_uuid`.

Пока POS не умеет надёжно классифицировать туриста и хранить доказательства,
туристическое исключение намеренно не включено. Для расчёта guard используется
вся сумма к оплате с чаевыми. Обе трактовки должны быть подтверждены специалистом
до production-sign-off.

## Uniform Format 1.31

Модуль должен формировать `INI.TXT` и `BKMVDATA.TXT`, после чего архивировать
`BKMVDATA.TXT` стандартным способом сжатия. Экспорт выбирает бизнес, путь и
период; создаёт уникальный идентификатор набора; завершает каждую запись CRLF;
нумерует записи `BKMVDATA.TXT` последовательно и соблюдает фиксированные длины.

| Record | Длина | Назначение для Kassa |
|---|---:|---|
| `A100` | 95 | начало набора |
| `B100` | 317 | бухгалтерские движения |
| `B110` | 376 | справочник счетов |
| `C100` | 444 | заголовок документа |
| `D110` | 339 | строки документа |
| `D120` | 222 | детали поступления/депозита |
| `M100` | 298 | движение/справочник запасов |
| `Z900` | 110 | конец набора и контрольные значения |

Для иврита спецификация задаёт Windows logical Hebrew ISO-8859-8-i либо DOS
CP862. Буквенно-цифровые поля дополняются пробелами справа, числовые — нулями
слева. `INI.TXT` содержит ведущую запись длиной 466 и summary-записи длиной 19
для каждого типа.

Регистрационный UI должен показать и дать распечатать:

- бизнес, путь, период, дату и время формирования;
- название ПО и номер регистрационного свидетельства;
- количество записей каждого типа;
- результат формирования и контрольную распечатку по разделу 2.6/приложению 4
  спецификации.

Архитектурно экспорт следует строить server-side из неизменяемых снапшотов,
версионировать по формату и покрыть golden-file тестами. Генерация только в
браузере неприемлема: старый WebView, кодировка, объём и прерывание вкладки не
дают достаточной надёжности.

## Israel Invoices в 2026 году

С 1 июня 2026 года инструкция устанавливает порог `5 000 ₪` без НДС для
применимых налоговых счетов: allocation number нужен, когда документ содержит
НДС, покупатель является зарегистрированным дилером, просит номер и сумма
превышает порог. Условия должны проверяться по официальной инструкции, а не по
одному числу в frontend.

Поле `allocation_no` уже есть, но этого недостаточно. Нужны:

1. server-side запрос к Tax Authority от имени уполномоченной стороны;
2. идемпотентный ключ и журнал запроса/ответа без утечки секретов;
3. сохранение allocation number в неизменяемом документе до его выдачи;
4. согласованное поведение при timeout, отказе и недоступности сервиса;
5. тестовая и production-авторизация.

## Release-gates

Production нельзя маркировать как legally compliant, пока одновременно не
выполнено следующее:

- [ ] письменный документ-аудит от израильского специалиста;
- [ ] Uniform Format 1.31 проходит официальный симулятор на требуемом объёме;
- [ ] получено требуемое свидетельство регистрации ПО;
- [ ] Israel Invoices работает для применимых документов и отказных сценариев;
- [ ] fiscal offline-flow согласован и проверен на реальном терминале;
- [ ] backup/restore и квартальная копия подтверждены операционным журналом;
- [ ] применены все production-миграции и пройдены frontend + pgTAP тесты;
- [ ] чеки на иврите, оригинал/копия/возврат и длинные документы проверены на
  Sunmi T2.
