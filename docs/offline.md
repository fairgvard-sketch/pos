# Офлайн-режим

Офлайн-слой нужен для продолжения кассового потока при кратком обрыве Wi‑Fi/4G,
timeout Supabase или перезапуске терминала. Он не превращает всё приложение в
полностью автономную систему: административные разделы и публичные сервисы всё
ещё требуют сеть.

## Что гарантируется

- приложение и горячие routes загружаются из PWA-кэша после первого запуска;
- каталог, точка, последняя смена и столы доступны из read-cache;
- продажа получает локальный номер `K-n` и временный чек;
- поддержанные мутации сохраняются в `localStorage` до успешного replay;
- повтор операции не дублирует заказ, оплату или строки;
- операции другой организации не отправляются под новой сессией устройства;
- привилегированная операция без PIN ждёт авторизацию, а не становится
  необратимой ошибкой.

## Из чего состоит слой

| Файл | Ответственность |
|---|---|
| `src/lib/offline/net.ts` | события сети, health probe, timeout и классификация ошибок |
| `src/lib/offline/types.ts` | типы операций и локальных заказов |
| `src/lib/offline/enqueue.ts` | постановка операций и создание local echo |
| `src/lib/offline/outboxStore.ts` | persist очереди, статусы, id map и временные номера |
| `src/lib/offline/drain.ts` | FIFO replay, backoff, auth wait и инвалидация кэша |
| `src/lib/offline/scope.ts` | изоляция по организации/точке/auth-user |
| `src/features/offline/OfflineOpsSheet.tsx` | состояние очереди и ручной разбор |

Телеметрия (`src/lib/telemetry.ts`, 074) в offline-слой не входит и живёт в
отдельной localStorage-очереди `kassa-telemetry`: ошибки и heartbeat МОЖНО
потерять, финансовые операции — нельзя. Телеметрия читает состояние outbox
для heartbeat, но никогда не пишет в него и не влияет на replay. Стоп очереди
(`failed`) дополнительно репортится как ошибка источника `outbox`.

## Read-cache

React Query сохраняет только данные, необходимые для продажи:

- `menu_categories`;
- `menu_items`;
- `modifier_groups`;
- `current_location`;
- `current_shift`;
- `tables`;
- `table_zones`.

Кэш хранится до семи дней, сбрасывается при изменении версии приложения и при
смене device scope. Заказы, отчёты и realtime-очереди целиком не персистятся.
Локальные заказы для бариста показываются отдельно из outbox store.

## Детекция сети

`navigator.onLine` используется только как быстрый сигнал. Реальная проверка —
`GET /auth/v1/health` Supabase:

- timeout probe — 5 секунд;
- повтор probe в offline — каждые 20 секунд;
- timeout боевой мутации продажи — 4 секунды;
- timeout одной replay-операции — 12 секунд;
- страховочный запуск drain — каждые 30 секунд.

`AbortController` не используется из-за старого WebView. Запрос, проигравший
`Promise.race`, может всё же дойти до сервера; поэтому idempotency UUID является
обязательной частью дизайна, а не оптимизацией.

## Поддержанные операции

Текущий outbox умеет:

| Kind | Действие |
|---|---|
| `order.place` | создать заказ у стойки |
| `order.pay` | оплатить новый или уже серверный заказ |
| `table.open` | открыть счёт стола |
| `table.append` | добавить позиции в счёт |
| `table.void` | отменить счёт |
| `table.discount` | изменить скидку счёта |
| `table.void_item` | отменить позицию |
| `queue.item_ready` | изменить готовность позиции |
| `queue.order_ready` | завершить заказ в очереди |

Настройки, каталог, склад, смены, возвраты, онлайн-заказы и бронирование не
следует считать offline-safe, пока для них отдельно не реализованы idempotency,
local echo и replay.

## Жизненный цикл операции

```text
pending ─► inflight ─► удалена после успеха
   │           │
   │           └─ сеть/timeout ─► pending + attempts + backoff
   ├─ доменная ошибка ──────────► failed
   ├─ нет staff-session ────────► blocked_auth
   └─ другой scope ─────────────► quarantined
```

- `failed` требует ручного retry или discard;
- `blocked_auth` автоматически возвращается в `pending` после PIN-входа;
- `quarantined` никогда не отправляется под текущей чужой сессией;
- `inflight` после rehydrate превращается обратно в `pending`.

Drain строго FIFO. Это сохраняет зависимости `place → pay` и `open → append`,
но означает, что одна `failed` или `quarantined` операция останавливает всю
очередь до ручного решения.

## Идентификаторы и зависимости

Каждая операция имеет:

- `id` — UUID server-side idempotency;
- `createdAt` — фактическое время действия пользователя;
- `orderKey` — локальный ключ группы зависимых операций;
- `orderId` — server UUID, если уже известен;
- `scope` — организация, точка и auth-user устройства.

После успешного `order.place` или `table.open` drain записывает соответствие
локального ключа серверному `order_id` в `idMap`. Следующая операция группы
получает серверный id через эту карту.

## Local echo и временный чек

При офлайн-продаже UI немедленно создаёт `LocalOrder`:

- номер вида `K-1`, который обнуляется по локальному дню;
- позиции и клиентский итог;
- временный receipt;
- состояние готовности для очереди бариста.

После синхронизации сервер возвращает настоящий `daily_number` и
`receipt_number`. Пользователь видит переход `K-n → #NN · чек №…`. Локально
fulfilled-заказ дополнительно отправляет `queue.order_ready`, чтобы после sync
не вернуться в очередь.

Временный номер не является фискальным номером. Источником истины остаётся
PostgreSQL sequence/counter.

Это остаётся блокером израильского fiscal sign-off: локальный документ нельзя
автоматически называть оригинальным фискальным документом до согласования
сценария с профильным специалистом. Cash guard применяется до постановки
операции в outbox и повторно внутри `pay_order` при replay. Полная матрица — в
[israel-compliance.md](israel-compliance.md).

## Scope-изоляция

Scope имеет форму:

```text
<org_id>:<location_id>:<supabase-auth-user-id>
```

Каждая новая операция штампуется текущим scope. При смене auth-сессии очищаются
безопасные локальные данные предыдущей точки:

- `kassa-query-cache`;
- `kassa-device-settings`.

`kassa-outbox` намеренно не удаляется: там могут находиться неотправленные
деньги. Чужая операция переводится в `quarantined` и остаётся для явного
разбора. Язык и стабильный UUID физического устройства также не сбрасываются.

Постановка новой операции без загруженного scope завершается ошибкой до записи
в outbox. Это fail-closed поведение: принадлежность финансовой операции нельзя
угадывать после смены аккаунта.

Отвязка терминала предупреждает о pending operations. Даже при принудительной
отвязке outbox не стирается молча.

Операции, созданные старой версией без поля `scope`, при восстановлении store
сразу получают `quarantined`. Автоматически отправлять их нельзя; решение
принимается вручную после проверки исходного аккаунта устройства.

## Staff-session при replay

`table.void`, `table.discount` и `table.void_item` требуют server-side
staff-session. Если приложение перезапустилось, PIN-store пуст, но outbox жив:

1. операция получает `blocked_auth`;
2. UI показывает ожидание PIN без красной доменной ошибки;
3. после входа store публикует новый session token;
4. drain снимает блокировку и продолжает FIFO.

Автор операции остаётся в payload как `staffId`, а право на replay подтверждает
сотрудник, вошедший в момент синхронизации.

С 086 это правило распространено на горячий поток: api-функции
place/pay/open/append/mark_* сами читают `currentStaffToken()` при каждом
вызове, поэтому replay несёт сессию сотрудника, актуального НА МОМЕНТ
синхронизации, а не на момент постановки в очередь. Сервер в мягком режиме:
без токена пропускает (хвост старой очереди), битый токен → `staff session
invalid` → `blocked_auth` → PIN → retry.

## Добавление новой offline-мутации

1. Добавьте payload и kind в `types.ts`.
2. Создайте enqueue helper и local echo/optimistic изменение.
3. Добавьте обработчик `runOp()` в `drain.ts`.
4. Определите зависимости через `orderKey`/`orderId`.
5. Добавьте server UUID и сохранение результата в `op_log`.
6. Решите, требуется ли staff token.
7. Инвалидируйте только связанные query keys после успеха.
8. Напишите тесты: online success, timeout после возможного server commit,
   restart из `inflight`, wrong scope, domain failure и PIN wait.

Без server-side идемпотентности операцию в outbox добавлять нельзя.

## Ручная проверка

- первый запуск онлайн, затем авиарежим и reload;
- продажа с печатью временного чека;
- возврат сети и получение реальных номеров;
- timeout после того, как запрос мог дойти на сервер;
- перезапуск между `place` и `pay`;
- привилегированная операция, закрытие приложения, повторный PIN;
- смена аккаунта устройства при непустом outbox;
- ручной retry/discard доменной ошибки.

Полный сценарий для терминала: [t2-smoke-test.md](t2-smoke-test.md).
