# Cardcom EMV — план интеграции

Цель: «Карта» в PaymentSheet сама отправляет сумму на пинпад — без
двойного ввода и ошибок сверки. Пока интеграции нет, «карта» — учётная
запись об оплате, проведённой на отдельном терминале.

## Что уже готово в кассе (046)

- `payments.provider / provider_ref / card_last4 / auth_code` —
  nullable-поля под реквизиты транзакции; учётная логика их не требует.
- Идемпотентность оплат: `pay_order(p_payment_uuid)` — повтор после
  обрыва сети вернёт первый результат. Тот же паттерн применим к
  запросам на терминал (transaction id генерирует касса).
- Правило возвратов «тем же способом» (030) — возврат картой пойдёт
  через терминал тем же каналом.

## Что нужно от кабинета Cardcom (получить до начала работ)

1. **Номер терминала** (מסוף) и **API Name / API Password**
   (Настройки → הגדרות API в личном кабинете Cardcom).
2. Какое железо: встроенный пинпад Cardcom (подключение к T2 по
   Bluetooth/USB?) или отдельный терминал с локальным API.
   От этого зависит канал:
   - **Cardcom Local API / SDK** — прямые вызовы к пинпаду из APK
     (надо запросить у поддержки Cardcom доки на локальный протокол);
   - **Cardcom Cloud (Low Profile / Indicator)** — через их сервер;
     для физического пинпада обычно НЕ подходит, это e-commerce канал.
3. Тестовый терминал (מסוף בדיקות) для разработки.

## Архитектура (когда будут креды)

```
PaymentSheet «Карта»
  → lib/cardcom.ts: charge({amount, txUuid})     // txUuid = p_payment_uuid
  → [канал: JS-мост APK → SDK пинпада | HTTP к локальному API терминала]
  → ответ approved {ref, last4, authCode} / declined {reason}
  → pay_order(..., p_payment_uuid: txUuid)       // + поля транзакции
```

- **Обрыв между списанием и pay_order** — главный кейс: деньги списаны,
  заказ не оплачен. Решение: txUuid пишется в outbox ДО вызова терминала;
  восстановление — запрос статуса транзакции по txUuid при старте.
- Статус-запрос по `provider_ref` — кнопка «проверить оплату» в
  журнале офлайн-операций.
- Возврат: `issue_refund` после успешного refund-вызова терминала,
  той же суммой, method='card'.
- НЕ хранить API-пароль во фронтенде: канал через локальный
  API терминала ок (LAN), облачные вызовы — только через Edge Function
  (Supabase) с секретом на сервере.

## Порядок работ

1. Получить креды + доки локального протокола (запрос в поддержку
   Cardcom: «חיבור קופה למסוף EMV, פרוטוקול מקומי»).
2. Прототип charge/refund/status в `lib/cardcom.ts` + мост в APK,
   если канал требует нативного SDK (пересборка APK).
3. PaymentSheet: панель «Карта» → статусы «Ждёт карту… / Одобрено /
   Отклонено», ручной фолбэк «провести без терминала» (текущее
   поведение) за настройкой.
4. Чек: последние 4 цифры + код авторизации в блоке оплаты.
5. Сверка: строка «карта (терминал)» в X/Z уже покрыта разбивкой
   method_gross (046).
