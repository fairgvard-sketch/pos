# База данных и Supabase

## Целевой проект

Боевой project ref: `qgmnxrgtlpyqglwqmsej`.

Перед миграциями и деплоем функций `scripts/check-project-ref.mjs` сверяет:

1. ref из `VITE_SUPABASE_URL` в `.env`;
2. `supabase/.temp/project-ref`, если CLI уже привязан.

Несовпадение останавливает команду. Guard появился после инцидента с применением
миграций не в тот проект и не должен обходиться ручным вызовом без проверки.

## Миграции

Схема хранится в `supabase/migrations/` и на текущем baseline состоит из
`001_foundation.sql` … `071_explicit_grants_baseline.sql`.

Правила:

- применять строго по номеру;
- не редактировать миграцию, которая уже могла попасть в окружение;
- любое исправление — новый файл со следующим номером;
- сначала миграция, затем frontend, который вызывает новый RPC;
- финансовые данные не удалять даже в миграциях исправления;
- SECURITY DEFINER функции всегда скоупить по организации и явно выдавать
  минимальные grants;
- каждый новый объект (таблица/функция) обязан получать ЯВНЫЙ `GRANT`
  нужным app-ролям — см. «Default privileges» ниже;
- вместе с новой миграцией обновлять `MIN_SCHEMA_VERSION` в
  `src/lib/schemaVersion.ts` — CI (`npm run check:schema`) требует равенства
  константы номеру последней миграции.

### Версия схемы для фронтенда (081)

`get_schema_version()` (SECURITY DEFINER, только `authenticated`) возвращает
максимальный номер из журнала миграций CLI
(`supabase_migrations.schema_migrations`) — отдельного счётчика, который можно
забыть обновить, нет. Фронт на старте рабочих экранов сверяет его с
`MIN_SCHEMA_VERSION`: отстающая база даёт экран «Требуется обновление базы
данных» (`SchemaGuard`), а не тихо пустой каталог. Ошибка сети/офлайн трактуется
как `unknown` и работу не блокирует — POS живёт по локальному кэшу.

## Default privileges: legacy production vs новые стеки

Новые стеки Supabase (локальный CLI, CI и свежесозданные проекты) поставляются
с ужесточёнными default privileges: объекты, созданные ролью `postgres` в
`public`, НЕ получают автоматических DML/EXECUTE грантов для
`anon`/`authenticated`. Действующий production (`qgmnxrgtlpyqglwqmsej`) создан
на старых дефолтах (полный DML + EXECUTE), поэтому исторические миграции с
паттерном «`REVOKE ... FROM anon, public`, полагаясь на неявный базовый грант»
там работают.

Следствия:

- pgTAP на чистом стеке ловит отсутствие грантов (так был найден красный CI,
  исправление devices/device-RPC — миграция 070);
- миграция 071 — baseline: механический перенос фактической модели доступа
  production (`supabase db dump`, фильтр authenticated/service_role, без
  anon — гостевые потоки идут через Edge Functions). Эквивалентность
  доказана диффом грантов свежего стека с production: 0 расхождений;
- каждый НОВЫЙ объект после 071 обязан выдавать явные GRANT в своей
  миграции — автоматики больше нет ни локально, ни на новых проектах.

Применение:

```bash
npm run check:ref
npm run db:push
```

Для локального полного прогона:

```bash
supabase start
supabase db reset
supabase test db
```

### Каталог

`stations`, `menu_categories`, `menu_items`, `item_variants`,
`modifier_groups`, `modifiers`, `menu_item_modifier_groups`.

Цены товаров, вариантов и модификаторов — целые агороты. `save_menu_item()`
сохраняет товар, варианты, привязки групп и упаковку (`p_supplies`, 075) одной
транзакцией. `reorder_menu()` атомарно обновляет порядок.

`variant_supplies` (075) — рецепт и упаковка: связка «товар/вариант →
расходник» (`qty` за единицу, `takeaway_only`). Продажа списывает расходники
в триггерах `order_items` по типу заказа (`takeaway`/`delivery` либо любой);
компенсации void/split возвращают ровно списанное по журналу `stock_movements`
(`order_item_id`), поэтому правки каталога между продажей и void не искажают
остатки. `variant_id` пересоздаются при каждом сохранении товара; клиент
адресует варианты позиционно (`variant_index`), а сохранение без `p_supplies`
(старый клиент) переносит variant-связки на новые варианты по имени.

Рецептуры (076): ингредиенты — те же `supply_items`, склад ведётся в
БАЗОВЫХ единицах (`г`/`мл`/`шт`) целыми числами, float запрещён (мешок муки
25 кг = 25000 г). Конвенция стоимости: для `г`/`мл` `supply_items.cost` —
агороты за 1000 базовых единиц (кг/л), для штучных — за единицу; сервер
стоимость не перемножает, конвенция живёт в UI (`costDivisor`).
`modifier_supplies` — расход модификатора (сироп 20 мл, овсяное молоко
180 мл), списывается триггером на `order_item_modifiers` без условия по типу
заказа; CRUD прямой (RLS, как у `modifiers`), `WITH CHECK` дополнительно
привязывает модификатор и расходник к организации. Дефолтные модификаторы
попадают в заказ автоматически, поэтому замена молока — это расход на
модификаторах группы «Молоко», а не рецепт товара. Компенсации те же:
журнал по `order_item_id` охватывает строки модификаторов без отдельного
кода. Лимиты `receive_stock`/`stock_take`/`add_waste` для `kind='supply'`
подняты под граммы (1 000 000 / 10 000 000 / 1 000 000), для товаров меню —
прежние.

## Tenant-изоляция и RLS

Каждая доменная таблица скоупится по `org_id`. Функции `auth_org_id()` и
`auth_location_id()` читают значения из JWT `app_metadata`. Клиент не выбирает
организацию параметром запроса и не может расширить scope фильтром.

Для `devices` модель строже на запись:

- чтение возможно в пределах организации для менеджерского списка;
- изменение своей строки связано с `auth.uid()`;
- `register_device` идемпотентен по `(org_id, device_uuid)`;
- `update_device_settings` обновляет только устройство текущего auth-user.

RLS — защита, а не вспомогательный UI-фильтр. Любой новый table/RPC должен
проверяться под двумя организациями. Базовые проверки находятся в
`supabase/tests/rls_scope.test.sql`.

## Двухуровневая авторизация

Устройство проходит Supabase Auth. `bootstrap_org()` создаёт initial data и
обновляет JWT metadata.

Сотрудник проходит `verify_staff_pin()`. Хеш PIN не читается клиентом благодаря
колоночным grants. Успешная проверка создаёт `staff_sessions`; токен передаётся
в привилегированные RPC как `p_staff_session`. `require_staff_perm()` проверяет
сессию, организацию, активность сотрудника и требуемый уровень права.

⚠️ ИЗВЕСТНЫЙ ДОЛГ (каталог мимо PIN): `menu_categories`, `modifier_groups`,
`modifiers`, `stations` правятся ПРЯМЫМИ запросами к таблицам, RLS там — только
`org_id = auth_org_id()` без PIN. Значит строгий режим `require_staff_perm` эти
таблицы НЕ защищает: любой авторизованный клиент своей org (в т.ч. любой
терминал) может создать/править/удалить категорию, группу модификаторов или
станцию мимо PIN-сессии. Для бэкофиса это удобно (веб-владелец пишет без RPC),
но как факт безопасности — дырка того же класса, что закрывал 090. Товары и
порядок так НЕ открыты: идут через `save_menu_item`/`reorder_menu` (единый
гейт). Осознанно не чиним сейчас; закрытие — отдельная миграция (перевод этих
таблиц на RPC с гейтом либо ужесточение RLS до owner-membership).

`require_staff_perm` работает в СТРОГОМ режиме (045, восстановлен 090): вызов
привилегированного RPC без токена сессии отклоняется. Был период регресса —
055 переопределила функцию телом 044 (правился фолбэк-CASE для `stock_take`) и
вернула мягкую `RETURN NULL` поверх строгой 045; на проде это держалось до 090.
090 берёт тело 055 и восстанавливает строгую NULL-ветку; `stock_take` как
менеджерский фолбэк сохранён. Инвариант закреплён в
`supabase/tests/staff_perm_strict.test.sql`, чтобы следующее переопределение
не откатило строгость молча. Горячий поток проверяет ОТДЕЛЬНУЮ функцию
`require_staff_session` и остаётся мягким (пара 044/045-зеркала ещё впереди).

### Throttle PIN и скоуп сессии по точке (095)

По итогам аудита безопасности закрыты две находки.

**Перебор PIN.** До 095 `verify_staff_pin` не считала неудачные попытки: ни
лимита, ни задержки. Практическое пространство — 10 000 (UI отправляет на 4-й
цифре), причём запрос сверяет хеш ЛЮБОГО сотрудника организации (`WHERE org_id
AND pin_hash`, `LIMIT 1`), поэтому с десятком сотрудников перебор схлопывается
ещё на порядок. Вызвать RPC мог любой с сессией устройства — включая украденный
или невозвращённый терминал. Теперь защита двухслойная и целиком в БД: `pg_sleep
0.3` на каждой неудаче (основной тормоз скриптового перебора) плюс блокировка
после 10 неудач за 15 минут. Скоуп счётчика — `(org_id, auth.uid())`, то есть
устройство: сотрудник неизвестен, пока PIN не сошёлся. Успешный вход журнал
обнуляет, поэтому опечатки бариста не копятся до блокировки. Порог щадящий
сознательно: блокировка кассы в час пик дороже, чем лишние попытки атакующему.
Журнал — таблица `pin_attempts` под RLS, без грантов клиенту; это сигнал
безопасности, а не аудит-трейл, и чистится свободно.

Throttle вынесен в хелперы `pin_throttle_check/_fail/_reset` и навешен на ОБЕ
функции, сверяющие PIN: `verify_staff_pin` (вход) и `punch_by_pin` (023, табель).
Это существенно: у табеля тот же bcrypt по тем же сотрудникам организации, и
защита только входа означала бы, что перебор просто переезжает на другой экран.
Порог живёт в одном месте — в `pin_throttle_check`.

Блокировка отдаётся отдельным кодом `pin_locked_out` → клиент показывает
«слишком много попыток», а не «неверный PIN»: иначе кассир долбит заведомо
мёртвый ввод верным кодом.

**Скоуп сессии по точке.** `require_staff_perm` сверяла `ss.org_id`, но НЕ
`location_id`, хотя колонка заполняется с 044. Токен точки А действовал на точке
Б той же организации, причём уровень права читается из настроек ТЕКУЩЕЙ точки
(`auth_location_id()`) — сотрудник точки со слабыми настройками получал право на
точке со строгими. 095 добавляет проверку и в `require_staff_perm`, и в
`current_actor_role` (условия обязаны совпадать, иначе «только владелец трогает
владельца» из 093 решалось бы по здесь-невалидному токену).

⚠️ Сессии, выданные ДО 095, могут иметь `location_id IS NULL` — они трактуются
как legacy-permissive, отвергается только явное расхождение. Иначе раскатка
разлогинила бы кассы посреди смены. Такие сессии истекают сами.

Оба инварианта закреплены в `supabase/tests/pin_throttle_session_location.test.sql`
(проверено мутацией: со снятым фиксом падают ровно эти три проверки).

### Веб-контур проверяет права (096)

До 096 `require_backoffice_or_staff` (091) принимала `p_perm`, но в ветке
веб-ролей его НЕ читала: членство в `organization_members` возвращало `staff_id`
сразу. `p_perm` проверялся только во второй ветке — кассовой. Значит
`locations.settings.perms` были правилами ТОЛЬКО кассы: владелец, пометивший
возвраты как `manager`-only, получал ограничение за кассой, но тот же человек
через веб-кабинет им связан не был. Один сотрудник имел разный объём прав в
зависимости от входа, а настройка молча не действовала там, где её разумно
считать действующей.

096 отображает веб-роль на уровень права так же, как кассовую:

- `owner` — выше ограничений, всегда;
- `manager` — как кассовый manager: проходит `all` и `manager`-уровни;
- `accountant` — только `all`-уровни; на `manage`/`refund` не допускается
  (роль читающая по замыслу 088).

**Почему `owner` — исключение.** Та же причина, что в 094 для кастомных ролей:
`perms` живут в `locations.settings`, и правит их сам владелец по праву
`manage`. Подчинив ему owner, получаем состояние, где владелец запирает сам себя
(выставил `manage='manager'`, связи со staff-строкой manager нет) и восстановить
доступ из продукта нечем. Это не гипотеза: `organization_members.staff_id`
опционален (091) — у владельца может вообще не быть роли сотрудника.

**Откуда берётся точка.** `organization_members` скоупится ОРГАНИЗАЦИЕЙ —
колонки `location_id` там нет, а в JWT веб-пользователя `auth_location_id()`
пуст. Поэтому `backoffice_perm_level` берёт точку из связанной staff-строки, а
при `staff_id IS NULL` возвращает БАЗОВЫЙ уровень: выбрать «какую-нибудь» точку
организации значило бы применить чужие правила.

Совместимость: на момент миграции в проде один активный член бэкофиса — `owner`,
веб-`manager`'ов нет, поэтому доступ никому не изменился. Правило зафиксировано
до появления второго веб-сотрудника, когда цена ошибки стала бы реальной.
Инварианты — `supabase/tests/backoffice_perm_enforced.test.sql` (проверено
мутацией: со снятой проверкой падают обе accountant-проверки).

⚠️ Остаётся в силе: опасные операции защищены НЕ только через `p_perm` —
`update_staff`/`set_staff_pin`/`delete_staff` (093) отдельно сверяют
`current_actor_role`, поэтому веб-менеджер не перевыпустит PIN владельца.

Горячий поток (`place_order`, `pay_order`, `open_or_get_table_order`,
`append_to_order`, `mark_item_ready`, `mark_order_ready`, `set_order_urgent`)
с 086 (087 для срочности) тоже принимает
`p_staff_session` и проверяет её `require_staff_session()` в МЯГКОМ режиме:
NULL пропускается (старые клиенты, хвост офлайн-очереди), переданный битый
токен даёт `staff session invalid`. Автор операции остаётся `p_staff_id` из
payload — сессия лишь подтверждает живого сотрудника за кассой. Прежние
реализации переименованы в `*_impl` и закрыты от клиентов (приём 068).
Строгий режим — отдельной миграцией после раскатки клиентов и опустошения
офлайн-очередей (зеркало пары 044/045).

## Веб-идентичности бэкофиса (088)

Владельческий бэкофис ANGLE (репозиторий `angles`, сборка в `/account/`) —
отдельный web-интерфейс поверх той же организации, а не экран POS. Доступ к
нему определяет `organization_members`: пара `(org_id, auth_user_id)`, роль
`owner|manager|accountant` и флаг `is_active`. PIN сотрудника сюда не
переносится — POS остаётся PIN-моделью, бэкофис использует Supabase Auth.

- `auth_backoffice_role()` — `SECURITY DEFINER`-хелпер: политики на
  `organization_members` не запрашивают рекурсивно ту же таблицу;
- `get_backoffice_context()` — bootstrap-payload (членство, организация,
  точки, счётчики). Организация берётся из JWT через `auth_org_id()`,
  клиентский org id не принимается; отсутствующее или деактивированное
  членство даёт `backoffice access denied`;
- членства не создаются из браузера: INSERT/UPDATE-грантов у `authenticated`
  нет, доступен только колоночный SELECT.

Начальное заполнение таблицы: миграция переносит во владельцы существующие
auth-аккаунты с валидным `org_id` в `app_metadata`, чтобы включение бэкофиса
никого не заблокировало. Сегодня это в том числе аккаунты терминалов — новых
данных им не открывает (их RLS-scope и так вся организация), но перед
появлением управляющих операций в бэкофисе нужно завести отдельный аккаунт
владельца и деактивировать членства устройств.

Инварианты закреплены в `supabase/tests/backoffice_members.test.sql`.

### Бэкофис (088, 089, 091)

- `get_backoffice_context` — bootstrap владельческого web-интерфейса: членство,
  организация, точки и счётчики. Требует активного членства в
  `organization_members`; org берётся из JWT;
- `sales_report` (089) — принимает владельца бэкофиса как альтернативу
  PIN-сессии: роль `owner`/`manager` в `organization_members` проходит без
  `p_staff_session`, остальные идут прежним путём через `require_staff_perm`.
  `accountant` владельческой ролью здесь не считается. Функция остаётся
  SECURITY INVOKER, поэтому чтение всё так же скоупится `orders_select` по
  `org_id` — членство даёт право на отчёт, но не расширяет видимость;
- `require_backoffice_or_staff(p_session, p_perm)` (091, ужесточён 096) — единый
  гейт ЗАПИСИ: сперва пускает члена бэкофиса без PIN, иначе — прежний
  `require_staff_perm`. С 096 веб-роль проверяется по уровню права
  (`backoffice_perm_level`), исключение — `owner`; см. раздел «Веб-контур
  проверяет права». Возвращает «автора» операции: `organization_members.staff_id`
  (модель Square — один человек, два входа; связь опциональна, при её
  отсутствии автор NULL). Веб-владелец адресует точку параметром, её
  принадлежность организации проверяет `assert_backoffice_location`
  (в JWT веб-владельца `location_id` нет);
- `patch_location_settings_web(p_location_id, p_patch, p_staff_session)` (091) —
  веб-версия настроек точки: логика merge идентична кассовой (064), но точка
  приходит параметром и право идёт через единый гейт. Кассовая
  `patch_location_settings` (по токену устройства) не тронута;
- `save_menu_item`, `reorder_menu` (092) — гейт заменён на
  `require_backoffice_or_staff`, поэтому веб-владелец правит меню без PIN.
  Меню org-scoped (скоуп по `auth_org_id`, не по точке), сигнатуры не менялись:
  касса зовёт те же функции, поведение с токеном идентично прежнему.
  ⚠️ Обе полностью пересинхронизируют варианты/группы — правка одного поля
  из веба обязана присылать текущие варианты обратно (иначе сотрёт), см.
  `backoffice/src/menu.js`.
- `create_staff`, `update_staff`, `set_staff_pin`, `delete_staff` (093) — гейт
  заменён на `require_backoffice_or_staff`, поэтому веб-владелец управляет
  командой без PIN. Отличие от меню: сотрудник привязан к точке, а в JWT
  веб-владельца `location_id` нет — `create_staff` прогоняет `p_location_id`
  через `assert_backoffice_location`. Касса `p_location_id` не шлёт, для неё
  проверка проходит молча и поведение не меняется;
- `save_role`, `delete_role`, `role_allows` (094) — кастомные роли: именованный
  набор прав (`roles.perms`) поверх базового `staff.role`. Роль — НАДСТРОЙКА,
  а не замена: `staff.role_id IS NULL` означает прежнее поведение байт-в-байт,
  откат = обнулить `role_id`. `require_staff_perm` получила ветку роли перед
  фолбэком на `locations.settings.perms`. Инварианты, включая регресс старого
  пути, закреплены в `supabase/tests/custom_roles.test.sql`.
  ⚠️ `'manage'` в набор роли не попадает (вырезается в `save_role`) и владелец
  ролью не ограничивается — иначе носитель роли выдал бы себе любые права,
  а владелец мог бы запереть сам себя. `verify_staff_pin` отдаёт `role_id` и
  `role_perms`, чтобы клиентский `can()` (`src/lib/perms.ts`) не расходился с
  сервером;
- `current_actor_role(p_session)` (093) — роль автора операции: членство в
  бэкофисе (веб) либо роль `staff` активной PIN-сессии (касса). На ней держится
  перенесённое в БД правило «владельца трогает только владелец»: не-владелец
  не может создать роль `owner`, повысить до неё, сменить PIN владельца или
  удалить его строку. До 093 это была ТОЛЬКО клиентская проверка
  (`StaffSection.canEdit`), то есть менеджер мог выписать себе owner-права
  прямым RPC. Инварианты закреплены в `supabase/tests/staff_backoffice_write.test.sql`.

### Телеметрия (074)

- `device_heartbeat` — лёгкий периодический апдейт собственной строки
  `devices`: версия приложения/моста печати, здоровье offline-очереди
  (`outbox_pending`, `outbox_oldest_at`, `outbox_failed`), `last_seen_at`.
  Heartbeat до `register_device` — тихий no-op: телеметрия не роняет кассу;
- `report_client_errors` — приём пакета ошибок с клиента: не более 20 за
  вызов, не более 100 новых fingerprint на устройство в день, все поля
  обрезаются на входе, повтор fingerprint наращивает `count`. Источники:
  `window|promise|react|outbox|print|shift` (082: `shift` — событие
  `shift_overdue`, когда открытая смена пересекла границу операционного
  дня `settings.shift.day_cutoff`, дефолт 04:00); неизвестный source
  клампится в `window`;
- операторские view `ops_fleet` и `ops_errors` — только для
  `service_role`/SQL Editor (см. [deployment.md](deployment.md),
  «Наблюдаемость парка»);
- `get_backoffice_fleet` (097) — веб-раздел «Девайсы» для владельца: парк
  своей организации (имя, точка, версии, здоровье offline-очереди,
  `last_seen_at`, `silence_seconds`). Гейт как у `sales_report` (089):
  членство владельца/менеджера ИЛИ `require_staff_perm(session,'manage')`;
  `SECURITY INVOKER`, чтение под RLS `devices_select` (чужая org недостижима).
  Стеки ошибок (`client_errors`) СЮДА НЕ ВХОДЯТ — остаются операторским
  каналом.

## Идемпотентность и время клиента

Миграция `042_offline_idempotency.sql` добавляет `op_log` и UUID-параметры для
денежных/составных операций. Повтор с тем же UUID возвращает первый результат,
а не создаёт второй заказ, платёж или строки.

Время, пришедшее от клиента, проходит через `clamp_client_ts()`: честное время
офлайн-операции сохраняется, но явно некорректные часы терминала не могут
безгранично исказить отчёты.

UUID должен создаваться до первой попытки запроса и повторно использоваться
после timeout. Новый UUID на retry разрушает гарантию идемпотентности.

## Edge Functions

| Функция | Назначение | Статус |
|---|---|---|
| `public-menu` | безопасная публичная витрина каталога | production |
| `public-order` | приём заказа и чтение статуса по client UUID | production |
| `public-reserve` | профиль точки, слоты, создание/отмена брони | production |
| `uniform-format-export` | набор Единого формата 1.31 за период (INI/BKMVDATA) | построена, деплой pending |
| `cardcom-payment` | будущая платёжная интеграция | карантин, `503`/`501` |

Публичные функции используют `service_role` только внутри Deno runtime и
возвращают allow-listed поля. Они не отдают гостю внутренние settings, staff,
RLS-токены или service key.

Деплой:

```bash
npm run functions:deploy
```

`cardcom-payment` нельзя считать готовой даже при наличии функции в проекте.
Текущий код намеренно не проводит оплату. См. [cardcom-plan.md](cardcom-plan.md).
