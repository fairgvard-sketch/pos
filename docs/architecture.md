# Архитектура Kassa

Этот документ описывает текущее устройство приложения. Обязательные продуктовые
и архитектурные инварианты заданы в корневом `AGENTS.md`; здесь показано, как
они реализованы в коде.

## Цели архитектуры

Архитектура оптимизирована не под универсальный ресторанный интерфейс, а под
быстрый поток кофейни:

- типовой заказ — товар → модификатор → оплата;
- кассир не ждёт сетевой ответ в горячем сценарии;
- бариста отмечает готовность одним тапом;
- закрытие браузера сбрасывает PIN-сессию сотрудника;
- финансовые операции переживают повтор запроса, перезапуск и обрыв сети.

## Общая схема

```text
React UI
  ├─ feature API ───────────────┐
  ├─ React Query read-cache     │ anon JWT устройства
  ├─ Zustand UI/session state   ▼
  └─ offline outbox ───────► Supabase
                              ├─ PostgreSQL + RLS
                              ├─ SECURITY DEFINER RPC
                              ├─ Realtime
                              └─ Edge Functions для публичных страниц

Canvas receipt ─► ESC/POS raster ─► APK JS bridge ─► Sunmi printer
                                  └► RawBT fallback
```

Браузерный клиент работает с Supabase только под `anon`-ключом. Организация и
точка берутся из JWT устройства. Публичные страницы не получают доступ к
внутренним таблицам: они вызывают узкие Edge Functions.

## Слои frontend

### Точка входа

`src/main.tsx` загружает полифиллы, выполняет capability-check и только после
этого монтирует React. Глобальный `AppErrorBoundary` ловит runtime-сбой всего
приложения.

`src/App.tsx` настраивает:

- React Query и персистентный read-кэш;
- React Router;
- детектор сети и replay офлайн-очереди;
- изоляцию локального состояния по scope;
- фоновую регистрацию устройства;
- route-level error boundary и lazy loading менеджерских экранов.

Горячие экраны `PIN`, `sell`, `hall` и `queue` входят в стартовый bundle.
Менеджерские и публичные экраны подгружаются лениво. `lazyWithRetry` один раз
перезагружает страницу, если после деплоя старая вкладка запросила уже удалённый
chunk.

### Feature-модули

Код организован по пользовательским сценариям в `src/features/`:

| Модуль | Ответственность |
|---|---|
| `auth` | привязка устройства, onboarding, PIN-вход и защита маршрутов |
| `sell` | корзина, скидки, чаевые, оплата и split-сценарии |
| `tables` | зал, столы, открытые счета и действия над ними |
| `queue` | realtime-очередь бариста |
| `receipt` | получение, canvas-рендер и печать чеков/тикетов |
| `shift` | открытие/закрытие смены, X/Z и кассовые движения |
| `timesheet` | вход/выход сотрудников и корректировки табеля |
| `transactions` | журнал операций и возвраты |
| `menu` | категории, товары, варианты, модификаторы и станции |
| `inventory` | остатки, приход, списание и инвентаризация |
| `loyalty` | гости, штампы и баллы |
| `online` | входящие и публичные онлайн-заказы |
| `reservations` | заявки, посадка и публичное бронирование |
| `reports`, `dashboard` | отчёты и мобильная сводка владельца |
| `settings` | настройки точки, сотрудников, устройства и тумблеры интерфейса POS (`settings.interface`) |
| `offline` | экран ручного разбора очереди |

Обычно feature содержит страницу/компоненты и `api.ts`. Общие примитивы лежат
в `src/lib/`, глобальные stores — в `src/store/`, доменные типы — в
`src/types/index.ts`.

## Авторизация и безопасность

### Устройство

Устройство входит через Supabase Auth по email и паролю. `bootstrap_org()`
создаёт организацию, точку и владельца, затем записывает `org_id` и
`location_id` в `app_metadata`. После onboarding JWT принудительно обновляется.

RLS получает scope через `auth_org_id()` и `auth_location_id()`. Это основная
граница tenant-изоляции. Во frontend нет и не должно быть `service_role`.

### Сотрудник

Сотрудник вводит четырёхзначный PIN. `verify_staff_pin()` проверяет bcrypt-хеш
в БД и возвращает профиль плюс токен серверной staff-сессии. Store
`kassa-staff-session` использует `sessionStorage`: новая вкладка или закрытие
приложения требуют повторного PIN.

Роли: `owner` > `manager` > `barista`. Видимость экранов проверяет клиент,
а привилегированные RPC проверяют staff-session через `require_staff_perm()`.
К ним относятся, в частности, возвраты, отмены, скидки, настройки, управление
сотрудниками и чувствительные складские/кассовые действия. Часть обычного
рабочего потока всё ещё доверяет авторизованному устройству — это осознанный
компромисс текущей модели.

## Состояние и кэши

| Данные | Механизм | Жизненный цикл |
|---|---|---|
| серверные reads | React Query | память; выбранные ключи сохраняются на 7 дней |
| корзина | Zustand | состояние приложения |
| PIN-сессия | Zustand persist | `sessionStorage`, до закрытия вкладки |
| язык | Zustand persist | `localStorage`, между сессиями |
| настройки терминала | Zustand persist | `localStorage` + фоновый merge в `devices.settings` |
| offline outbox и local echo | Zustand persist | `localStorage`, до синхронизации/разбора |
| Supabase device session | `supabase-js` | локальная auth-сессия с refresh token |

В offline read-cache входят только `menu_categories`, `menu_items`,
`modifier_groups`, `current_location`, `current_shift`, `tables` и
`table_zones`. Версия
приложения используется как cache buster.

Per-device sync отправляет локальные изменения в БД с debounce. При наличии
локального snapshot он выигрывает конфликт; после очистки scoped localStorage
store восстанавливается из `devices.settings`. Синхронизация повторяется после
`SIGNED_IN`, reconnect и ручного retry в настройках устройства.

## Поток продажи

Онлайн-путь:

1. клиент создаёт UUID заказа;
2. `place_order` валидирует каталог и снапшотит цены, НДС, скидку и итоги;
3. `pay_order` получает отдельный UUID оплаты и создаёт неизменяемые платежи;
4. сервер присваивает номер чека;
5. UI инвалидирует связанные queries и печатает чек.

Если запрос не ответил за четыре секунды или сеть недоступна, операция попадает
в outbox. Интерфейс создаёт локальное эхо заказа и продолжает работу. Детали —
в [offline.md](offline.md).

## Надёжность UI

- мутации используют optimistic update или offline fallback;
- Query mutations настроены с `networkMode: always`, чтобы offline-ветка не
  зависала в `pending`;
- сетевые и доменные ошибки разделяются;
- PWA service worker кэширует bundle и навигационный fallback;
- route-level и app-level error boundaries заменяют белый экран понятной ошибкой;
- менеджерские chunks ленивые, горячий путь остаётся в стартовом bundle;
- capability-screen рендерится без React, если движок заведомо непригоден.

## Деньги и аудит

`Agorot` — целое число. Любая цена, скидка, чаевые, оплата, возврат и остаток
хранятся без float. Форматирование и разбор пользовательского ввода выполняет
`src/lib/money.ts`.

Итоги заказа являются снапшотом. После оплаты нельзя пересчитать исторический
чек из текущих цен каталога. Финансовые записи не удаляются: исправление
представляется компенсирующей операцией с причиной, автором и временем.

## UI, локализация и RTL

Интерфейс поддерживает русский и иврит. Каждый экран задаёт `dir`, а layout
использует логические направления `start/end` и `ms/me`. Фискальный чек
рендерится отдельно и печатается как растр, поэтому порядок иврита не зависит
от ESC/POS-кодировки принтера.

Новый UI должен соблюдать 8pt-сетку, сенсорные цели не меньше 44 px, WCAG AA,
сдержанную серую палитру и существующие классы из `src/index.css`.
