# Kassa v2 — правила работы с проектом

Kassa — POS для кофеен, пекарен и specialty coffee. `legacy/` используется
только как продуктовый референс; импортировать оттуда код напрямую нельзя.

## Документация проекта

Перед выполнением любой задачи:

1. Прочитать `AGENTS.md`.
2. Использовать `README.md` как индекс актуальной документации.
3. Прочитать относящиеся к задаче документы:
   - `docs/architecture.md` — архитектура, модули и маршруты;
   - `docs/development.md` — разработка, UI-правила и тестирование;
   - `docs/database.md` — Supabase, RLS, RPC и миграции;
   - `docs/offline.md` — offline-cache, outbox и replay;
   - `docs/android-printing.md` — Android, WebView и печать;
   - `docs/deployment.md` — выпуск, эксплуатация и откат;
   - профильные `docs/online-orders.md`, `docs/reservations.md`,
     `docs/cardcom-plan.md` и `docs/t2-smoke-test.md`.

Не дублировать в этом файле меняющийся снимок routes, миграций и готовых фич.
Фактический источник истины — текущий код и последовательность миграций. При
расхождении с документацией явно сообщить об этом и обновить соответствующий
документ вместе с исправлением.

В итоговом отчёте ссылаться на документы, повлиявшие на решение. При изменении
поведения, архитектуры, API, схемы/RPC, offline, печати или deployment обновлять
документацию обязательно.

## Главный продуктовый принцип

Скорость рабочего процесса — приоритет №1:

- типовой заказ: не более трёх тапов — товар → модификатор → оплата;
- мутации в горячем потоке optimistic-first, UI не ждёт сеть;
- экран бариста: один тап означает «готово», без подтверждения;
- PIN отправляется автоматически после ввода четвёртой цифры;
- POS проектируется под палец и работу в спешке, а не под desktop-формы.

## Архитектурные инварианты

1. Деньги — только целые агороты через `src/lib/money.ts`. Float запрещён.
2. Финансовые записи не удаляются и не переписываются. Void/refund/discount —
   компенсирующая операция с аудитом.
3. Во frontend допускается только Supabase anon key. `service_role` существует
   только в серверных Edge Functions.
4. Каждая доменная таблица скоупится по `org_id`; tenant-защита обеспечивается
   RLS через JWT `app_metadata`.
5. Сумма, НДС, скидки, названия и цены снапшотятся в момент операции.
6. Денежные и составные мутации идемпотентны: UUID создаётся до первой попытки
   и повторно используется после timeout/replay.
7. Составные изменения выполняются атомарным RPC, а не цепочкой клиентских
   запросов, способной частично завершиться.
8. Уже применённые миграции не редактировать. Исправление — новый следующий
   номер и forward-only изменение.

## Авторизация и безопасность

- Устройство — аккаунт Supabase Auth. `org_id`/`location_id` приходят из JWT.
- Сотрудник — PIN-сессия в `sessionStorage`; закрытие приложения блокирует POS.
- Привилегированные RPC принимают `p_staff_session` и проверяют права в БД.
- Навигационные role guards не являются заменой RLS или server-side permission.
- PIN hash, токены, service keys и персональные данные не логировать.
- Cardcom Edge Function остаётся в карантине до полной реализации безопасного
  server-verified потока; клиентской сумме и redirect доверять нельзя.

## Работа с данными

- Reads — через feature `api.ts` и React Query.
- Локальное UI-состояние — компонент; Zustand только для межэкранного/persist.
- `locations.settings` изменять через `patch_location_settings`.
- Товар, варианты и modifier links сохранять через `save_menu_item`.
- Порядок каталога изменять через `reorder_menu`.
- Для offline-мутации обязательны server idempotency, typed outbox operation,
  local echo, replay handler и тесты timeout/restart/wrong-scope.
- Не удалять outbox при logout, сбросе кэша или обработке runtime-ошибки.

## UI и RTL

- Светлый минимализм, без ярких градиентов и эмодзи.
- Использовать глобальные классы из `src/index.css`.
- Отступы по 8pt-сетке; предпочтительно 4/8/12/16/24/32 px.
- Touch target горячего потока — не меньше 44 px.
- Основной текст — `text-gray-900`, вторичный — минимум `text-gray-500`.
- Иерархия задаётся размером и весом, не декоративным цветом.
- Для RTL использовать `dir`, `start/end`, `ms/me`; не `ml/mr`.
- Не использовать `backdrop-blur` в часто открываемых POS-overlays.
- Новый i18n-ключ обязан существовать и в `ru`, и в `he`.

## Целевое устройство

Основное железо — Sunmi T2 Mini, Android 7.1, 1920×1080 landscape, принтер
80 мм. Legacy-бандл транспилируется для Chrome 52, но это не гарантия runtime:
фактическую совместимость определяет capability-check и ручной тест на реальном
терминале. Не заявлять поддержку версии WebView без пройденного
`docs/t2-smoke-test.md`.

Печать: canvas → ESC/POS raster → APK bridge; RawBT/browser — fallback. Изменение
нативного моста или production origin требует нового APK.

## Проверка изменений

Минимум перед завершением задачи:

```bash
npm run lint
npm run test:run
npm run build
npm run check:bundle
```

Для DB/security изменений дополнительно:

```bash
supabase start
supabase db reset
supabase test db
```

Изменения горячего потока, offline, WebView или печати требуют ручного
`docs/t2-smoke-test.md`. Нельзя объявлять физическую печать или совместимость
терминала проверенной только по unit tests.
