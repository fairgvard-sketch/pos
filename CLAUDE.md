# Kassa v2 — POS для кофеен, пекарен и specialty coffee

Перестройка с нуля (июль 2026). Старая ресторанная версия лежит в `legacy/` —
**только как референс** (модификаторы, лояльность, print-agent, i18n); не импортировать из неё код напрямую.

## Главный принцип

**Скорость рабочего процесса — приоритет №1.** Каждый экран проектируется от количества тапов:
- Типовой заказ: ≤3 тапа (товар → модификатор → оплата)
- Optimistic UI на все мутации — интерфейс никогда не ждёт сеть
- Экран бариста: 1 тап = готово, без диалогов подтверждения
- PIN-переключение сотрудника ≤2 сек (автоотправка на 4-й цифре)

## Стек

- **Frontend**: React 19, TypeScript, Tailwind CSS 3, React Query v5, Zustand v5
- **Backend**: Supabase (PostgreSQL, RLS, Realtime)
- **Сборка**: Vite 8

## Архитектурные инварианты (не нарушать)

1. **Деньги — целые агороты** (`src/lib/money.ts`). Никаких float. Конвертация в ₪ только при отображении.
2. **Никогда не удалять финансовые записи.** Void/refund/скидка — это новая запись, не UPDATE/DELETE. Аудит-трейл священен.
3. **Только anon-ключ во фронтенде.** service_role не существует для клиента. RLS — реальная защита, не декорация.
4. **org_id везде.** Каждая доменная таблица имеет org_id; RLS скоупит через `auth_org_id()` из JWT app_metadata. Схема готова к SaaS, работаем на одной точке.
5. **Totals снапшотятся в заказ** (сумма, НДС, скидки) в момент операции — не вычисляются заново из связей.
6. **Мутации идемпотентны** — client-generated UUID на операциях. Это заготовка под offline-очередь (фаза 7).

## Модель авторизации (двухуровневая)

1. **Устройство** = аккаунт Supabase Auth (email+password, вводится один раз при настройке кассы).
   `org_id`/`location_id` прописаны в `app_metadata` JWT функцией `bootstrap_org()` → их читает RLS.
2. **Сотрудник** = PIN внутри приложения. `verify_staff_pin()` (SECURITY DEFINER) сверяет bcrypt-хеш,
   PIN не покидает БД, `pin_hash` не читается клиентом (колоночные гранты).
   PIN-сессия живёт в sessionStorage (`authStore`) — закрыл браузер → касса заблокирована.

Известный компромисс: DB доверяет устройству (JWT), персональная роль сотрудника
enforced на клиенте. Ужесточение (staff-scoped токены) — после MVP.

## Доменная модель (кофейня, не ресторан)

Заказ — позиция в **очереди у стойки**, не стол:
- `daily_number` (#42 на сегодня), `type` (here/takeaway), `customer_name`
- Статусы: `open → paid → fulfilled` | `voided`
- Экран бариста — realtime-очередь по `paid`-заказам
- Номер чека — sequence в PostgreSQL (не в памяти агента!)

## Дизайн-стандарты (соблюдать во всём новом UI)

Индустриальные правила, не вкусовщина. Проверять при написании каждого экрана.

1. **Сетка 8pt.** Все отступы/размеры/gap кратны 4, лучше 8: используем Tailwind-шкалу
   `1 2 3 4 6 8 10 12 16` (=4/8/12/16/24/32/40/48/64px). Избегать «съезжающих»
   `px-5`/`py-5` (20px) — брать `4` или `6`. `gap-3`/`p-4`/`p-6` — норма.
2. **Сенсорные мишени ≥44px** (WCAG/Apple; Material 48). POS = тач-планшет, палец в спешке.
   Интерактивные элементы (кнопки товаров, степперы ±, чипы, пункты меню) — высота
   ≥44px (`h-11`+). Мелкие иконки-кнопки ✎/✕ в админке — исключение (мышь, не касание).
3. **Контраст текста (WCAG AA ≥4.5:1).** Основной текст (названия, цены, суммы) —
   `text-gray-900`. Вторичный/подписи — минимум `text-gray-500` (не `gray-400`:
   он ~2.8:1, проходит только для третьестепенных лейблов).
4. **Иерархия — размером и весом, не цветом.** Заголовок крупнее+жирнее, вторичное
   мельче+серее. Шкала шрифта только Tailwind (`text-xs/sm/base/lg/xl/2xl`).
5. **Правило 60-30-10.** 60% фон (белый/`#eceef1`), 30% панели (серый), 10% акцент
   (тёмная CTA `bg-gray-900`). Не разбавлять акцентными заливками.
6. **Близость (гештальт).** Отступ ВНУТРИ группы меньше, чем МЕЖДУ группами.
7. **Внешний отступ ≥ внутреннего** — контейнер от края не теснее, чем контент внутри.

## Текущий контекст работы (июль 2026)

**Целевое устройство:** T2 Mini POS Terminal (Sunmi-класс). Экран 11.6", **1920×1080
ландшафт**, **Android 7.1**, встроенный термопринтер 80мм, 2D-сканер, NFC, Wi-Fi, 4G.
**Касса работает на нём в бою** через APK-обёртку.

**Развёртывание на T2 (работает, проверено):**
- Прод: Vercel, стабильный домен `pos-self-sigma.vercel.app`, деплой пушем в `main`.
- **APK-обёртка** `android/`: WebView + JS-мост `window.KassaAndroid` → тихая печать
  ESC/POS на встроенный принтер (официальный `com.sunmi:printerlibrary`). Сборка:
  GitHub Actions «Android APK» → артефакт `kassa-sunmi-apk`. APK пересобирать только
  при изменении моста; UI/логика обновляются деплоем Vercel.
- **Android 7.1 совместимость решена**: `@vitejs/plugin-legacy` (chrome>=52,
  nomodule-бандл) + полифилл `crypto.randomUUID` (`src/lib/polyfills.ts`).
- Печать чека (`ReceiptSheet.handlePrint`), приоритет: мост APK → RawBT → браузер.
  Чек рендерится в canvas (`printCanvas.ts`) → ESC/POS растр (`lib/escpos.ts`).
- PWA `display: standalone` (НЕ fullscreen — иначе на терминале пропадает
  системная навигация и кассу не свернуть).

**Настройки per-device** — таб «Касса» (localStorage, `deviceStore`): автоблокировка,
PIN после продажи, звук оплаты, способ печати. Идёт адаптация идей из Square POS
(юзер скидывает скрины — разбирать и брать полезное).

**Возможные следующие шаги:** Cardcom; лояльность v2 (возврат баллов при refund,
лояльность на счетах столов, QR-карта); офлайн v2 (скидка/void-item офлайн-строк
стола, правка payload pending-append).

**Офлайн-слой (фаза 7, миграция 042 + `src/lib/offline/`):**
- Работают без сети: counter-продажа (place+pay в очередь, временный чек
  «מסמך זמני» с локальным номером K-n, фискальный номер присваивается при
  синке) и столы (открыть/дозаказ/оплатить/отменить через локальное эхо).
  Онлайн-only: возвраты, item-split, move/merge, смена (open/close/cash),
  лояльность, админка/табель — блок с подсказкой `offlineBlockedHint`.
- Сервер: `op_log` (op_uuid→result, в той же транзакции) — replay любой
  операции возвращает ПЕРВЫЙ результат (pay_order не тратит второй номер
  чека); `clamp_client_ts` — честное время операции (`p_paid_at`/`p_placed_at`,
  daily_number по дню продажи). `p_payment_uuid` шлётся и онлайн.
- Клиент: `outboxStore` (localStorage `kassa-outbox`, строгий FIFO,
  failed-голова СТОПит очередь до ручного retry/discard), `drain.ts` (движок
  replay, сетевые ошибки → backoff, доменные → failed), `net.ts`
  (`withOfflineFallback` — таймаут 4с уводит продажу в офлайн; проба
  /auth/v1/health каждые 20с; Promise.race, без AbortController — Chrome 52),
  `enqueue.ts` (типизированные постановщики + LocalOrder-эхо).
- Read-кэш: `PersistQueryClientProvider` (localStorage `kassa-query-cache`,
  ключи каталог/точка/смена/столы, 7д, buster `__APP_VERSION__`) — холодный
  старт офлайн работает. Бейдж — `OfflineBadge` в сайдбаре, журнал —
  `features/offline/OfflineOpsSheet` (+ строка в Операциях и Настройки→Оплата).
- Гард: закрытие смены заблокировано при непустой очереди (replay должен
  попасть в открытую смену). Эхо-заказы видны в очереди бариста (K-n) и в
  Операциях до синка; дедуп по client_uuid/serverOrderId.

**Уже сделано (не переделывать):** чек-хешбонит (иврит/RTL, `features/receipt/`),
фиск. нумерация (020), реквизиты (019), сплит + смешанная оплата (021), модуль зала
